# 08 — 批判性自审：v3 的六大缺陷

> 每次迭代都要带着批判看上次的内容。这篇文档就是那个批判。

## 缺陷 1：UX 北极星丢失了

**用户的核心目标是"更好的用户体验"。** 这是用户原话。

但 v3 七份文档读下来，OS 隐喻占据了全部心智空间。你记住了 Kernel、Scheduler、Watchdog——但你还记得这个系统在服务谁吗？

| 文档 | 提到 UX 的次数 | 占比 |
|------|---------------|------|
| 01-kernel-userspace | 0 | 0% |
| 02-scheduler | 0 | 0% |
| 03-memory | 0 | 0% |
| 04-watchdog | 0 | 0% |
| 05-supervisor | 0 | 0% |
| 06-journal | 0 | 0% |
| 07-blueprint | 0 | 0% |

**一个关于"如何做出好产品"的体系，七份文档里 UX 这个词一次都没出现。**

这是一个严重的结构性缺陷。OS 隐喻提供了强大的工程框架，但它遗漏了整个系统中最重要的角色：**终端用户**。

在真正的 OS 中，用户通过 GUI/Shell 与系统交互。在 AI Product OS 中，用户是谁？他们在哪里？他们如何感知"这个产品好不好"？

**需要补充：** UX 作为 AI Product OS 的"Display Server"——用户感知层。Gate 拦截了坏产出，但谁来定义"好体验"？

## 缺陷 2：零可执行代码

1380 行文档。0 行可以运行的代码。

一个 PM 读完 v3 后的状态：
- 知道了 MLFQ 调度的原理 ✓
- 知道了 Circuit Breaker 三态模型 ✓
- 能在明天上班时搭建任何一个子系统 ✗

**这不是工具书，这是科幻小说。** 它描述了一个完美的系统但没告诉你怎么迈出第一步。

### 具体缺失

- L1 Schema Gate 的 JSON Schema 长什么样？
- L2 Policy Engine 的正则规则怎么写？
- L3 LLM Judge 的 Prompt 模板是什么？
- Capability Token 的生成和验证代码？
- Journal 的 append-only 存储怎么实现（10 行 SQLite？）

**需要补充：** 每个子系统的最小可行实现（MVP code），让读者能在 10 分钟内跑起来。

## 缺陷 3：没有 Bootstrap 路径

v3 的三级部署模型描述了"如果你已经有一个完美的 OS 该怎么部署"。

但它没回答一个更基本的问题：**这个 OS 本身是怎么被创建出来的？**

类比：Linux 能从 Kernel 0.01 进化到 6.x，是因为 Linus 先写了一个能 boot 的最小内核。v3 的文档相当于在描述 Linux 6.x 的完整架构——对于一个想从零开始的人来说毫无用处。

**需要补充：** Day 1 Bootstrap 指南。第一个可用的 Gate 怎么搭？（答案可能是：一个 Makefile + 3 个 Python 脚本。）

## 缺陷 4：反馈机制的正确性未被验证

用户的核心问题是"反馈机制怎么设计才能保证反馈本身正确"。

v3 的回答是：三角验证 + 定期校准 + 假阳性审计。

**问题是：这些校准方法本身从未被校准过。** 这是一个无限递归问题——谁来看守看守者？

v3 对此没有给出满意的回答。它假设"人工校准"是最终的锚点。但人工也会犯错、疲劳、有偏见。

**需要补充：** 反馈机制的"最终锚点"到底是什么？是用户行为数据（转化率/留存率）？是 A/B 测试的统计显著性？是用户访谈的定性洞察？需要明确：**在什么条件下，我们相信反馈是正确的？**

## 缺陷 5：子系统交互未定义

v3 描述了 6 个独立子系统，但子系统之间的接口是模糊的。

| 交互 | 问题 |
|------|------|
| VERIFY -> WATCHDOG | VERIFY 发现的异常如何触发 WATCHDOG 熔断？阈值是什么？ |
| JOURNAL -> MEMORY | JOURNAL 的记录何时、如何被提炼为 MEMORY 的知识？ |
| SCHEDULER -> SUPERVISOR | SCHEDULER 决定 kill 一个 Agent，SUPERVISOR 如何响应？ |
| WATCHDOG -> CAPABILITY | WATCHDOG 检测到越权，是拦截还是事后告警？ |

文档 07 有一个交互矩阵，但它是静态的。没有描述数据如何在这些子系统之间流动，没有定义接口契约（API），没有描述竞态条件（如果 VERIFY 和 WATCHDOG 同时对一个 Agent 做出冲突的判断怎么办？）。

## 缺陷 6：OS 隐喻的边界未被审视

OS 隐喻是有力的，但它也在一些地方产生了误导：

| OS 概念 | AI PM 强行映射 | 哪里不 fit |
|---------|---------------|-----------|
| 进程隔离 | Agent 上下文隔离 | OS 进程是完全隔离的。AI Agent 之间需要协作和共享上下文。过度隔离会杀死协作。 |
| 抢占式调度 | Sev-0 抢占其他任务 | OS 抢占是微秒级的。AI 任务的"抢占"意味着丢弃当前工作——成本远高于 CPU context switch。 |
| 文件系统 | Journal | 文件系统存储的是字节。Journal 存储的是决策。后者不可比较、不可 diff——这使得"回滚"比文件系统回滚复杂得多。 |
| Signal | SIGKILL | OS Signal 到达是即时的。对 Agent 发 SIGKILL 意味着它正在进行的 LLM 调用被浪费了。 |

**隐喻的价值在于解释力，但隐喻被当作蓝图时会引入系统性误差。**
