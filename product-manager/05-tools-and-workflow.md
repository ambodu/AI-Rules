# 工具与工作流

| 工具 | 用途 | 关键能力 |

## 二、AI PM 的周度工作流

```
周一 ─ 上传项目 Brief → AI 生成结构化 PRD → PM 审查、优化、批准
周二 ─ PRD 进入开发流水线 → AI 开始生成代码
周三 ─ PM 审查初始产出 → 对照 AC 检查
周四 ─ 迭代：收紧偏离的 Spec、确认达标的产出、标记技术审查
周五 ─ 定义下周工作 + 回顾本周评测数据
```

### 关键变化

- **状态会议失去意义** — 产出在几天内可见，不需要开同步会
- **协调成本下降** — 定义和审查成为全部工作
- **压缩反馈循环** — 从"几周后发现需求有问题"变成"几天内发现需求有问题"

## 三、代码审查流程（PM 视角）

PM 不做代码审查（那是工程师的事）。PM 做**产品意图审查**：

### PM 审查清单

1. **意图匹配** — 输出是否符合我们想要的？
2. **边缘情况** — 指定的边缘情况是否都被处理了？
3. **用户目标** — 是否真正服务于用户的目标（而非仅仅"技术上正确"）？
4. **范围控制** — AI 是否偷偷加了不该加的东西？
5. **体验一致性** — UI/交互是否符合 Design System？

### 信任建立机制

| 做法 | 为什么 |
|------|--------|
| PR 保持小规模（3-7 屏的 diff） | 容易审查，风险低 |
| 亲手写 PR 标题和描述 | AI 总结"做了什么"，人需要"为什么这么做" |
| 提交前自查 | 展现人类努力，提前发现明显问题 |
| 附带测试的截图/录屏 | 比"代码看起来对"更可信 |
| 标注 AI 使用范围 | 信号：哪些经过了人工审查，哪些是纯 AI 产出 |

## 四、验证流程

### 验证的三层把关

```
AI 代码产出
  │
  ├── 第 1 关：自动验证（必过）
  │   · 单元测试是否通过？
  │   · Lint/类型检查是否通过？
  │   · 视觉回归测试是否通过？
  │
  ├── 第 2 关：AI 验收 Agent（独立模型）
  │   · 是否满足 AC？
  │   · 是否引入回归？
  │   · 是否有安全风险？
  │
  └── 第 3 关：人工审查
      · 产品意图是否正确？
      · 用户体验是否优秀？
      · 战略是否对齐？
```

### 验证防作弊

AI 能生成伪造的测试结果来让自己的代码"看起来正确"。防御措施：
- 生成和验证用**不同的模型**
- 自动测试在**隔离环境**中运行
- 写操作/删除操作/生产系统改动需要**更严格的验证**

## 五、反馈数据管道

```
用户接触点（产品内反馈 / 客服工单 / 应用商店评论 / NPS 调查 / 行为数据）
        │
        ▼
    聚合层（LLM 自动分类、聚类主题、情感分析）
        │
        ▼
    优先级排序（严重度 × 频率 × 影响用户数 × 战略价值）
        │
        ▼
    路由分发（Prompt 优化 / 模型微调 / 设计调整 / 知识库更新）
        │
        ▼
    验证闭环（A/B 测试 → 指标确认 → 全量）
        │
        ▼
    记忆写入（防止同类问题复发）
```

## 六、评估 AI 产出效率的指标

| 指标 | 计算方式 | 意义 |
|------|----------|------|
| 设计漂移率 | 非标准组件页面占比 | 一致性健康度 |
| 返工率 | 修复 AI 代码的时间 / 首次原型时间 | AI 产出质量 |
| 缺陷逃逸率 | 合并后发现的 Bug 比例 | 验证有效性 |
| 审查疲劳度 | AI 建议接受率 | >85% 说明审查失效 |

## 七、自检清单

- [ ] 是否有完整工具栈覆盖定义→评测→反馈→验证？
- [ ] PM 是否在做产品意图审查而非代码审查？
- [ ] 验证是否分层（自动 → AI → 人工）？
- [ ] 反馈数据管道是否闭环？
- [ ] 是否监控审查疲劳度和返工率？
