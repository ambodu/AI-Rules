# 反馈机制设计

## 核心问题

> 什么样的反馈机制设计，可以保证反馈本身正确，最终保证产品正确？

## 一、反馈机制的层级模型

AI 产品研发的反馈系统分三个层级：

```
Model（引擎）
  └─ Harness（车间：工具、权限、沙箱）
      └─ Loop（排班与验收制度：运行机制、反馈循环、质量判定）
          └─ Memory（记忆：记录错误 → 诊断原因 → 验证 → 提炼规则 → 复用）
```

Loop 的核心逻辑：
> 模型跑一轮 → 从环境里收到反馈 → 对照标准检查 → 没达标就带着反馈再跑一轮 → 直到验收通过

## 二、反馈闭环的三大环节

### 第 1 环：信号收集

| 信号类型 | 来源 | 示例 |
|----------|------|------|
| 显性反馈 | 用户主动行为 | 评分、纠错按钮、文字评价、投诉 |
| 隐性反馈 | 用户行为数据 | 停留时长、跳出率、是否转人工、重复操作 |
| 专业反馈 | 内部标注/审查 | 安全审查、品牌一致性检查、专家标注 |
| 系统反馈 | 自动化检查 | 测试结果、Lint 错误、性能指标、幻觉检测 |

### 第 2 环：信号处理

信号必须转化为可执行的动作：

1. **分类标注** — 将用户反馈归类为结构化数据（功能性缺陷 / 体验问题 / 内容错误 / 安全风险）
2. **严重度分级** — 对齐用户影响：

| 严重度 | 定义 | 用户影响 |
|--------|------|----------|
| Sev-0 | 答案看起来正确但实际错误 | 侵蚀信任（最危险） |
| Sev-1 | 答案明显错误，用户无法恢复 | 挫败用户 |
| Sev-2 | 答案明显错误，用户可通过重新表述恢复 | 惹恼用户 |

3. **路由** — 不同问题走不同修复路径：
   - Prompt 层问题 → Prompt 优化
   - 模型能力问题 → SFT/RLHF
   - 产品设计问题 → 交互/流程调整
   - 知识缺失问题 → RAG 知识库更新

### 第 3 环：效果验证

- **离线验证**：在评测集上验证修复效果
- **灰度验证**：小范围上线，观察核心指标
- **A/B 测试**：对比新旧版本在真实场景下的表现
- **持续监控**：幻觉率、用户满意度、解决率等指标追踪

## 三、独立验收机制（最关键设计原则）

> 运动员不能兼任裁判。模型不能给自己打分。

### 为什么必须独立验收？

1. **自我评估偏差** — 模型倾向于给自己的工作打高分
2. **幻觉验证** — AI 可以生成伪造的测试结果和虚构的单元测试输出来让代码"看起来"正确
3. **上下文污染** — 执行 Agent 的上下文包含大量假设和推断，在相同上下文中评估会继承这些偏差

### 验收 Agent 设计

```
执行 Agent（写代码/做设计）
        ↓ 产出
验收 Agent（独立评审）
        ↓ 判定
通过 → 进入下一步
不通过 → 带着具体反馈回执行 Agent
```

验收 Agent 的关键要求：
- **干净上下文** — 不接触执行过程的中间推理
- **明确标准** — 基于预设的评测 Rubric，而非"感觉"
- **独立模型** — 使用不同模型进行验证（如 Claude 生成代码，GPT 做审查）
- **可解释判定** — 给出具体的通过/不通过理由，而非二分类标签

## 四、反馈频率与节奏

| 层级 | 频率 | 反馈内容 |
|------|------|----------|
| 即时反馈 | 每次输出 | 自动检查（Lint/测试/格式） |
| 短期反馈 | 每日/每任务 | AI 验收 Agent 评估 |
| 中期反馈 | 每周 | PM 意图审查 + 用户测试 |
| 长期反馈 | 每月/每版本 | 产品数据分析 + 战略对齐检查 |

## 五、反馈质量保障

确保反馈机制本身正确的方法：

1. **反馈信号三角验证** — 任一判断需要至少两个独立信号源确认
2. **反馈校准** — 定期让人工评估与 AI 评估做对比，计算一致率
3. **反馈衰减检测** — 监控是否出现了"审查疲劳"（AI 建议接受率 >85% 意味着审查失效）
4. **假阳性/假阴性审计** — 定期抽查通过的产物是否存在漏检缺陷，拒绝的产物是否误杀
5. **反馈优先级** — Sev-0（看起来对但实际错） > Sev-1（明显错且阻塞） > Sev-2（明显错但可恢复）

## 六、典型反馈回路流程

```
1. 用户遇到问题 → 提交反馈 / 行为异常被检测
2. 系统自动分类并评估严重度
3. 路由到对应处理流程：
   ├─ 内容错误 → 知识库更新 → RAG 验证 → 灰度发布
   ├─ 行为错误 → Prompt 优化 → 离线评测 → A/B 测试
   ├─ 设计缺陷 → PM 评审 → 重设计 → 用户测试
   └─ 安全风险 → 立即下线 → 安全团队介入
4. 修复后 → 验收 Agent 评审 → 灰度验证 → 全量
5. 关闭反馈 → 写入 Memory（防止复发）
```
