# 02 — Scheduler 调度器：任务治理

## OS 类比

OS Scheduler 回答三个问题：
1. **谁先运行？**（优先级）
2. **运行多久？**（时间片）
3. **什么时候强制切换？**（抢占）

AI PM 的 Scheduler 同样回答三个问题：
1. **什么任务先做？**（需求优先级 + 缺陷严重度）
2. **AI 做多久该交出结果？**（时间/预算/迭代上限）
3. **什么时候强制停止并升级？**（止损 + 人工介入）

## 一、MLFQ 调度算法 → AI 任务调度

### OS 中的 MLFQ (Multi-Level Feedback Queue)

```
Priority 0 (最高): 交互式任务 ← 用户正在等待
Priority 1:       短批处理    ← 快速完成的任务
Priority 2 (最低): 长批处理    ← 后台任务

规则:
- 新任务从 Priority 0 开始
- 用完时间片未完成 → 降级到下一级
- 被抢占（有更高优先级任务到达）→ 回到原来级别
- 定期 Boost：所有任务回到 Priority 0（防止饥饿）
```

### AI PM 中的 MLFQ

| 优先级 | AI 任务类型 | 时间片 | 抢占规则 |
|--------|-----------|--------|----------|
| **P0** | Sev-0 缺陷修复 | 30 min | 任何任务被 P0 抢占 |
| **P1** | Sev-1 缺陷 / 本周核心需求 | 2 hours | P0 可抢占 P1 |
| **P2** | Sev-2 缺陷 / 下个迭代需求 | 4 hours | P0/P1 可抢占 |
| **P3** | 探索性任务 / 技术债 | 无保证 | 总是可抢占 |

### 饥饿防护

定期 Boost（每周一早上）：所有任务回到最高优先级重新评估。防止低优先级任务永远得不到执行。

## 二、Zombie Reaping：僵尸回收

### OS 中的 Zombie Process

进程已终止但父进程未调用 `wait()`，进程描述符仍占用内核内存。僵尸进程不消耗 CPU 但消耗资源槽位。

### AI PM 中的 Zombie Task

| 僵尸类型 | 表现 | 回收策略 |
|----------|------|----------|
| **无限循环 Agent** | 反复尝试同一修复但永远不达标 | 3 轮后自动 kill + 人工介入 |
| **停滞子任务** | 子 Agent 被 spawn 后无响应 | 超时检测 + 级联 kill |
| **孤儿产出** | AI 生成了但无人审查的代码 | 48h 未审查 → 自动关闭 |
| **僵尸需求** | 创建后再无更新的 Ticket | 7 天无活动 → 自动关闭或降级 |

### 回收实现

```
Scheduler 后台任务（每 5 分钟）：
  1. 扫描所有活跃 Agent
  2. 检测：
     · 同一操作失败 3 次以上 → kill + alert
     · 单次运行超过时间片 2x → kill + alert
     · 子 Agent 无心跳超过 5 min → kill parent + cascade
     · 产出等待审查超过 48h → notify PM + auto-close
  3. 释放被占用资源（Token 配额、并发槽位、审查带宽）
```

## 三、Admission Control：准入控制

### OS 类比

当系统负载过高时，OS 拒绝新进程创建（`fork()` 返回错误）。这防止系统整体崩溃。

### AI PM 中的准入控制

| 资源 | 硬限制 | 软限制（触发告警） | 超限行为 |
|------|--------|-------------------|----------|
| 并发 Agent 数 | 10 | 7 | 新任务排队 |
| 每日 Token 预算 | $50 | $35 | 降级使用更便宜模型 |
| 审查带宽 | 20 产出/天/PM | 15 | 新产出排队等待审查 |
| 修复循环总数 | 50 轮/天 | 35 | 优先级重排 |

**关键**：准入控制不是限制生产力，而是保护系统稳定性。就像 OS 不会因为"用户还想开更多程序"就让系统崩溃。

## 四、Context Switch 开销 → 反馈成本

### OS 中的 Context Switch

切换进程有开销：保存寄存器、刷新 TLB、加载新页表。频繁切换会消耗大量 CPU。

### AI PM 中的 Context Switch

每次 PM 切换审查上下文都有开销：
- 重新理解 AC
- 重新建立用户场景心智模型
- 重新加载相关代码/设计的上下文

| 场景 | Context Switch 成本 | 优化策略 |
|------|---------------------|----------|
| 审查同功能的多轮迭代 | 低 — 上下文热 | 批量审查 |
| 审查不同功能 | 中 — 需切换场景 | 按功能分组审查时间段 |
| 审查 + 写 Spec | 高 — 模式切换 | 定义和审查分开不同时段 |
| 中断式审查请求 | 极高 — 破坏 Flow | 异步审查队列，非实时响应 |

### 批处理优化

```
不是：随时收到 AI 产出 → 立即审查（高频 Context Switch）

而是：
  上午 9-11：定义（写 Spec + AC）— 单一上下文
  下午 2-3：审查产出（批量处理，同一功能族）— 减少切换
  下午 4-4:30：Eval 会议 — 回顾 + 更新体系
```

## 五、Real-Time vs Batch 分类

来自 Agent-OS Blueprint (Koubaa, 2025) 的延迟分类：

| 类别 | 延迟要求 | AI PM 实例 | Scheduler 策略 |
|------|----------|-----------|---------------|
| **HRT** (硬实时) | 立即响应 | Sev-0 缺陷、安全漏洞、隐私泄露 | 抢占一切、专用 Agent |
| **SRT** (软实时) | < 30 min | Sev-1 缺陷、核心功能阻断 | 高优先级、有 Deadline |
| **DT** (延迟容忍) | 无 Deadline | 功能开发、优化、探索 | 公平共享或 FIFO |

**HRT 的 EDF (Earliest Deadline First) 调度**：Sev-0 = Deadline "现在"，Sev-1 = Deadline "今天"，P0 需求 = Deadline "本周"。

## 六、抢占式多任务 → 迭代中止

### OS 中的抢占

高优先级任务到达时，OS 强制暂停当前任务，保存状态，切换到高优先级任务。

### AI PM 中的抢占

| 抢占事件 | 响应 |
|----------|------|
| Sev-0 缺陷被发现 | 当前正在修复的 Sev-2 任务被抢占，Agent 状态保存，转去修复 Sev-0 |
| 用户反馈激增（某个功能被大量投诉） | 该功能相关的所有 Agent 任务优先级提升到 P0 |
| 成本超预算 | 所有 P3 任务暂停，P1/P2 降级使用更小模型 |
