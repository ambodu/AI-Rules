# 04 — Watchdog 看门狗：运行时安全与反馈

## OS 类比

Watchdog Timer 是 OS 最底层的安全保障：
- 系统定时"喂狗"（kick the watchdog）
- 如果系统挂起/死循环，"狗"不再被喂
- Watchdog 超时 → 强制重启系统

AI PM 同样需要多层 Watchdog：熔断器、预算执行器、循环检测器、异常信号注入器。

## 一、Circuit Breaker：防止连锁故障

### OS 类比

电路过载时熔断器断开，防止整个系统烧毁。熔断后不是永久断开——它会间歇性探测，确认故障恢复后自动重连。

### AI PM 中的 Circuit Breaker

来自 Oxios/Namzu 的三态模型：

```
            (错误次数超阈值)
  CLOSED ───────────────────────► OPEN
  (正常)                           (停止接受新任务)
    ▲                                  │
    │                                  │ (冷却时间到)
    │                                  ▼
    └────────(探测成功)─────── HALF-OPEN
                                 (试探性接受 1 个任务)
                                    │
                                    │ (成功 → CLOSED)
                                    │ (失败 → OPEN，冷却时间翻倍)
```

### 触发条件

| 熔断器 | 触发条件 | OPEN 后行为 | 冷却时间 |
|--------|----------|-----------|----------|
| **模型故障** | 3 次连续 5xx 错误 | 切换到备用模型 | 30s → 60s → 120s (指数退避) |
| **幻觉率暴涨** | 10 次产出中 > 30% 含幻觉 | 所有产出强制加 L3 人工确认 | 直到幻觉率回归正常 |
| **审查带宽饱和** | PM 审查队列 > 20 个 | 拒绝新 AI 任务，仅处理 Sev-0 | 直到队列 < 10 |
| **成本超预算** | 日消耗 > 日预算 120% | 所有 Agent 降级到便宜模型 | 到次日预算重置 |

### 降级策略（Graceful Degradation）

当熔断器 OPEN 时，系统不应完全不可用：

```
模型不可用 →
  主线路: Claude Opus (不可用)
  降级 1: Claude Sonnet (性能略降但可用)
  降级 2: GPT-4o (不同提供商)
  降级 3: 缓存结果 (之前成功过的相同查询)
  最终降级: "此功能暂时不可用，我们的团队已被通知" (宁可拒绝也不要错误结果)
```

## 二、Limit Checker：预算执行

### OS 类比

Linux Cgroups 限制进程组的 CPU/内存/IO 使用量。超限时进程被 throttle 或 kill。

### AI PM 中的 Limit Checker

每次 Agent 行动前检查，超限即停：

| 预算维度 | 检查点 | 超限行为 |
|----------|--------|----------|
| **Token 预算** | 每次 LLM 调用前 | 拒绝调用，Agent 被 kill |
| **成本预算** | 每次 LLM 调用前 + 调用后确认 | 拒绝调用 |
| **时间预算** | 每次迭代前 | 超时 → kill + 输出当前最佳结果 |
| **迭代预算** | 每次 Loop 前 | 达到上限 → kill + 人工介入 |
| **并发预算** | fork() 时 | 拒绝新 Agent，排队等待 |

### 预算配置示例

```
任务: 修复用户报告的 Bug (Sev-1)
├─ Token 预算: 50K tokens (输入+输出)
├─ 成本预算: $1.50
├─ 时间预算: 30 min
├─ 最大迭代: 5 轮
└─ 最大并发子 Agent: 3

任务: 探索性功能开发 (P3)
├─ Token 预算: 200K tokens
├─ 成本预算: $5.00
├─ 时间预算: 4 hours
├─ 最大迭代: 20 轮
└─ 最大并发子 Agent: 5
```

## 三、Loop Detector：循环检测

### OS 类比

死锁检测：OS 检测进程间的循环等待（A 等 B, B 等 C, C 等 A），检测到后强制打破循环。

### AI PM 中的循环检测

| 循环模式 | 表现 | 检测方式 | 打破方式 |
|----------|------|----------|----------|
| **修复死循环** | Agent 反复修同一个 Bug 但永远不达标 | 相同的修复动作 × 3 次 | kill + 人工介入 |
| **验收死循环** | 验收 Agent 和 执行 Agent 来回传递但不收敛 | 3 轮后评分未改善 | 强制升级到人工 |
| **需求循环** | 需求被反复修改永远不定 | AC 被改 5 次以上 | 锁定 AC + 人工仲裁 |
| **探索漂移** | Agent 在探索任务中不断扩展范围 | 产出与原始需求的距离持续增大 | 范围硬重置 |

### 循环打断规则

```
if (same_failure_pattern == 3):
    → 停止自动化流程
    → 将完整的 Loop 历史 + 验收记录打包
    → 人工介入：判定是 AC 问题 / 模型能力问题 / 产品设计问题
    → 根据判定重新开始（修正了 AC 后）或关闭任务（承认当前模型做不到）
```

## 四、信号注入：异步打断

### OS 类比

`SIGINT` (Ctrl+C)、`SIGSTOP`、`SIGKILL` — 用户可以随时向进程发送信号，强制中断或终止。

### AI PM 中的信号机制

| 信号 | 触发条件 | Agent 行为 |
|------|----------|-----------|
| **SIGPAUSE** | PM 手动暂停 | 保存当前状态，等待恢复 |
| **SIGSTOP** | 成本/时间超限 | 保存状态 + 输出已有结果 |
| **SIGKILL** | Sev-0 出现，需立即释放资源 | 立即终止，不保存状态 |
| **SIGREDIRECT** | PM 发现方向错了 | 丢弃当前工作，带新方向重启 |
| **SIGCHECK** | PM 想看看"你现在在做什么" | 输出当前进度摘要 + 下一步计划 |

### 自动信号

除了人工发送，Watchdog 自动发送：

```
Watchdog 自动 SIGPAUSE:
  · 检测到循环（连续 3 轮无改善）
  · 幻觉率 > 阈值（可能是模型本身有问题）
  · 操作模式可疑（短时间内大量写操作）

Watchdog 自动 SIGKILL:
  · 检测到明确的危险操作（rm -rf /, DROP TABLE）
  · 输出中包含被 Policy Engine 标记的禁止内容
  · 成本在 5 分钟内飙升 500%
```

## 五、Health Check：持续心跳

### OS 类比

Systemd/Launchd 定期检查服务是否存活。未响应 → 重启。

### AI PM 中的 Health Check

| 检查项 | 频率 | 不健康阈值 | 自动修复 |
|--------|------|-----------|----------|
| Agent 心跳 | 30s | 2 min 无响应 | kill + 重启 |
| 验收 Agent 健康 | 每小时 | 一致率 < 80% | 触发重新校准 |
| Eval 覆盖率 | 每天 | 核心场景覆盖率 < 90% | 告警 + 建议新增 Eval |
| 反馈闭合率 | 每天 | < 70% | 告警 + 增加审查资源 |
| 审查队列深度 | 实时 | > 20 | 降速新任务生成 |

## 六、Watchdog 的层次

```
硬件 Watchdog（不可绕过）
  └─ 由外部系统提供，独立于 AI 系统
     例：Kubernetes liveness probe, CloudWatch alarm

OS 级 Watchdog（Kernel 内）
  └─ L1/L2 Gate, Policy Engine, Capability Check
     不可被 AI Agent 影响

应用级 Watchdog（Agent 可感知但不能修改）
  └─ Circuit Breaker, Limit Checker, Loop Detector
     Agent 知道自己被监控

用户级 Watchdog（人工）
  └─ PM 的 SIGCHECK, 周度 Eval 会议
     最后的防线
```
