# 03 — Memory Management：知识与上下文管理

## OS 类比

OS 内存管理解决的核心问题：
1. **分层存储** — Register → L1 Cache → L2 → RAM → SSD → HDD，越靠近 CPU 越快但越小
2. **虚拟内存** — 每个进程有自己的地址空间，不互相干扰
3. **页面置换** — 内存不够时把最少使用的页换出到磁盘
4. **Copy-on-Write** — fork() 时父子共享内存，只在真正写入时才复制

AI PM 的 Memory Management 解决同样的问题：Agent 的上下文如何分层、如何隔离、如何在"内存"不够时优雅降级。

## 一、三层上下文：Hot / Warm / Cold

来自 AgentRM (arXiv:2603.13110) 和 Oxios 的设计：

```
Tier 0 (Hot) ──── 当前工作上下文 ──────── 完全保真，快速访问
  │ 容量: ~8K tokens                      （当前对话/当前任务）
  │ 延迟: < 100ms
  ▼
Tier 1 (Warm) ─── 近期上下文 ──────────── 自适应压缩
  │ 容量: ~100K tokens                    （本周相关任务/知识）
  │ 延迟: < 1s
  ▼
Tier 2 (Cold) ─── 长期知识库 ──────────── 序列化存储，按需召回
    容量: 无上限                          （历史决策/失败模式/产品知识）
    延迟: < 10s
```

### 各层的 AI PM 内容

| 层级 | 存储内容 | 更新频率 | 过期策略 |
|------|----------|----------|----------|
| **Hot** | 当前 AC、当前对话、当前任务的上下文 | 每轮更新 | 任务完成后清空 |
| **Warm** | 本周 Eval 结果、进行中的功能上下文、最近失败模式 | 每日 | 7 天未访问降级 |
| **Cold** | 产品知识库、历史决策记录、完整 Eval 历史、用户反馈归档 | 每周 | 季度审查 + 淘汰 |

### 页面置换策略（LRU → AI PM）

OS 中 LRU (Least Recently Used)：内存满时换出最久未访问的页。

AI PM 中：Hot 层满时，最久未引用的上下文被压缩并降级到 Warm 层。Warm 层满时降级到 Cold 层。

**保护位机制**：频繁被引用的上下文（如核心 AC、正在修复的缺陷描述）设置保护位，不被置换。

## 二、虚拟内存 → 上下文隔离

### OS 中的虚拟内存

每个进程有自己的地址空间，进程 A 的地址 `0x1000` 和进程 B 的地址 `0x1000` 映射到完全不同的物理内存。进程之间天然隔离。

### AI PM 中的上下文隔离

| 隔离维度 | 问题 | 方案 |
|----------|------|------|
| **Agent 间隔离** | Agent A 的上下文被 Agent B 错误继承 | 每个 Agent 独立的上下文空间 |
| **任务间隔离** | 上一个任务的推理影响下一个任务 | 任务结束后清空 Hot 层 |
| **用户间隔离** | 用户 A 的数据泄漏给用户 B | 用户级 Capability + 数据隔离 |
| **功能间隔离** | 功能 A 的设计约束被功能 B 违反 | 功能级设计规则分库 |

### 实现模式

```
Agent A 的地址空间：
  ├─ 当前任务上下文（Hot）← 仅 Agent A 可见
  ├─ 功能 A 设计规则（Warm）← 功能 A 相关 Agent 共享
  └─ 产品全局知识库（Cold）← 所有 Agent 只读共享

Agent B 的地址空间：
  ├─ 当前任务上下文（Hot）← 仅 Agent B 可见
  ├─ 功能 B 设计规则（Warm）← 功能 B 相关 Agent 共享
  └─ 产品全局知识库（Cold）← 同上
```

**关键**：Agent A 不能"看到" Agent B 的 Hot 层。就像进程不能访问其他进程的内存。

## 三、Copy-on-Write → 高效分支

### OS 中的 CoW

`fork()` 时父子进程共享同一块物理内存。只有当其中一个真正要写入时，才复制那一页。这避免了不必要的全量复制。

### AI PM 中的 CoW

当需要并行验证同一个产出时：

```
主验证 Agent fork 出 2 个子验证 Agent：
  ├─ 子 Agent 1：从安全视角审查（共享主 Agent 的上下文）
  └─ 子 Agent 2：从 UX 视角审查（共享主 Agent 的上下文）

CoW 优化：
  · 开始时三者共享同一份 AC + 产出的上下文
  · 只有当某个子 Agent 需要额外的上下文时（如安全策略库），才加载
  · 避免了每次 fork 都复制完整的 AC 和上下文
```

## 四、Memory Leak → 上下文污染

### OS 中的内存泄漏

程序分配了内存但从不释放，随着运行时间增长，可用内存逐渐减少直到系统崩溃。

### AI PM 中的上下文泄漏

| 泄漏类型 | 表现 | 检测 | 修复 |
|----------|------|------|------|
| **决策残留** | 已废弃的设计假设仍在影响后续决策 | 定期审查 Warm 层中引用的"事实"是否仍成立 | 过期标记 + 定期清理 |
| **Eval 膨胀** | 评测集只增不减，包含大量已不相关的 Case | Eval 通过率接近 100%（没有区分度） | 定期淘汰"永远通过"的 Eval |
| **规则腐烂** | 针对已修复的 Bug 添加的规则仍在使用，产生误报 | 假阳性率上升 | 规则有效性审查 |
| **上下文堆积** | Agent 的 Hot 层没有在任务完成后清理 | Agent 行为越来越保守/混乱 | 强制任务边界 + 上下文重置 |

### 垃圾回收 (GC)

定期（每周）运行 GC：
1. 标记：哪些上下文/规则/Eval 在过去 30 天被引用过？
2. 清除：未引用的 → 降级到 Cold 或删除
3. 压缩：合并重复或高度相似的规则

## 五、遗忘曲线 → 知识衰减

### 类比

OS 不关心内容本身的价值。但 AI PM 需要。Kleos 引入了 Ebbinghaus 遗忘曲线：知识随时间衰减，但每次召回会强化。

### AI PM 中的知识衰减

```
产品决策的"记忆强度" = f(距今时间, 被引用次数, 决策重要性)

衰减规则：
├─ 7 天内未被引用的决策 → 强度 -20%
├─ 30 天内未被引用的决策 → 强度 -50%，从 Warm 降级到 Cold
├─ 90 天内未被引用的决策 → 强度 -90%，归档
└─ 被引用的决策 → 强度 +30%（强化），保持在 Warm

保护规则：
├─ 安全相关决策 → 永不过期
├─ 合规相关决策 → 永不过期
├─ 被 3 次以上引用的决策 → 半衰期延长 2x
```

### 定期"做梦"：后台知识整合

Kleos 的六阶段"梦境循环"在空闲时运行。AI PM 同样需要：

```
每周五下午 4:30（Eval 会议后）：
  1. Replay — 重放本周关键决策
  2. Merge — 合并相关知识（"这两个 Bug 根因相同"）
  3. Prune — 删除不再相关的规则
  4. Discover — 发现新模式（"这类问题本周出现了 5 次"）
  5. Decorrelate — 分离错误关联
  6. Resolve — 解决矛盾（"这个 Eval 和那个 Eval 冲突了"）
```
