# 产品正确性保障

## 核心问题

> 怎样设计产品来保证正确性？如何确保最终结果一定是更好的用户体验？

## 一、产品正确性的三层防线

### 第 1 层：定义正确性（上游）

在你开始构建之前，先回答：

1. **谁是裁判？** — 谁来判断产品是否正确？（用户？PM？独立 Agent？组合？）
2. **标准是什么？** — 什么构成"正确"？（精确的验收标准 + 可量化的指标）
3. **证据是什么？** — 需要什么来证明正确性？（测试结果？用户行为数据？A/B 数据？）
4. **失败模式是什么？** — 当不正确时会发生什么？（用户流失？错误决策？安全风险？）

> 你不能评估的东西，你就不应该构建。

### 第 2 层：验证正确性（中游）

从三个维度持续验证：

**代码层验证（确定性）**
- 单元测试 / 集成测试 / E2E 测试
- Lint / 类型检查 / 合约测试
- 设计 Token 一致性检查

**AI 层验证（概率性）**
- LLM-as-Judge：独立模型按 Rubric 打分
- 幻觉检测：事实一致性校验
- 安全性审查：注入攻击 / 有害内容检测

**人工层验证（权威性）**
- 高敏感决策人工审批
- 品牌 / 语气 / 价值观一致性检查
- 边缘 Case 定期审计

### 第 3 层：监控正确性（下游）

上线不是结束，而是大规模测试的开始。

- **设计漂移率** — 使用了非标准 Token/组件的页面比例
- **缺陷逃逸率** — 合并后发现 Bug 的比例
- **用户挫败信号** — 重复操作 / 跳出 / 转人工比率
- **幻觉率** — 输出中事实错误的比例
- **首问解决率** — 用户问题在首次交互中被解决的比率

## 二、为正确性而设计产品

### 原则 1：AC 是可执行的功能规格，不是沟通辅助

传统开发中，Accept Criteria 是"开发者之间的沟通工具"。AI 时代，AC 是**直接被执行的功能规格**。

差的 AC：「用户可以筛选结果」

好的 AC：
```
筛选面板
- 位置：搜索结果列表上方，粘性定位
- 筛选项：类别（多选）、价格范围（滑块）、评分（单选）
- URL：筛选参数反映在 query string 中，支持分享
- 空态：无匹配结果时显示"尝试放宽条件"引导
- 加载态：骨架屏，非空白 loading
- 错误态：网络失败时保留上次结果 + 重试按钮
- 移动端：底部抽屉面板，含"应用"和"重置"按钮
```

### 原则 2：设计即是约束

好的产品设计通过**缩小可能性空间**来保障正确性：
- Design System → 限制组件选择，防止 UI 漂移
- 模板化交互 → 限制交互模式，确保一致性
- 预设输出格式 → 限制 AI 输出结构，降低不确定性
- 类型系统 → 在编译阶段就排除一大类错误

### 原则 3：失败可恢复

在产品设计中内置恢复机制：
- 操作可撤销（Undo）
- 版本可回滚
- AI 输出附置信度标记
- 关键操作二次确认
- 用户总是可以回到"安全状态"

### 原则 4：渐进式暴露复杂度

不要让用户一次性面对 AI 的全部能力。按用户成熟度逐步开放：
1. 新手模式 → 有限选项，高引导
2. 熟练模式 → 更多控制
3. 专家模式 → 完整 API / 自定义

### 原则 5：校准而非自动化

AI 产品设计的目标不是 100% 自动化，而是找到**最优的人机协作点**：
- 哪些决策 AI 可以自动执行？（确定性高、风险低）
- 哪些需要 AI 建议 + 人工确认？（风险中等）
- 哪些必须人工决策？（高风险、涉及价值观/法律）
- 哪些应该纯人工？（品牌定义、战略决策）

## 三、质量飞轮

基于 AI 产品质量持续改进框架：

```
         ┌──────────────────┐
         │  Offline Quality  │  ← 上线前：参考示例、指标、Rubric
         └────────┬─────────┘
                  ↓
         ┌──────────────────┐
         │  Online Quality   │  ← 上线后：用户显/隐性信号监控
         └────────┬─────────┘
                  ↓
         ┌──────────────────┐
         │Continuous Improve │  ← 采样失败 Case → 发现新失败模式
         └────────┬─────────┘
                  ↓
         ┌──────────────────┐
         │ Update Eval/Metric│  ← 更新指标 & 参考数据 & 修复
         └────────┬─────────┘
                  ↓
              (回到顶部)
```

生产环境会超越你所有的最佳努力。无论上线前的指标多好，实际使用中总会出现意外。

## 四、评测金字塔

不同成本和可靠性的验证方法分层使用：

```
         ┌─────────┐
         │ 人工审查  │  ← 最可靠，最昂贵（用于校准和高风险决策）
        ┌┴─────────┴┐
        │ LLM 评审  │  ← 可扩展，需定期与人工结果校准
       ┌┴───────────┴┐
       │ 代码级检查   │  ← 确定性、快速、便宜（运行在所有层次之上）
      └───────────────┘
```

- **代码级检查**（底层）：所有产出必过
- **LLM 评审**（中层）：大部分产出由 AI 评审
- **人工审查**（顶层）：抽样审计 + 高风险决策 + 校准基准

## 五、用户体验正确性指标

| 北极星指标 | 定义 | 为什么重要 |
|-----------|------|-----------|
| 任务完成率 | 用户成功完成任务的比例 | 直接度量产品价值 |
| 首问解决率 | 首次交互解决用户问题的比例 | AI 产品核心效率指标 |
| 用户满意度 (CSAT) | 用户主观满意度评分 | 体验质量的综合反映 |
| 再次使用率 | 用户回访比例 | 信任与价值的代理指标 |

| 隐藏指标 | 定义 | 为什么重要 |
|----------|------|-----------|
| 事实准确性 | 输出内容与事实一致的比例 | 短期难量化，长期跌破则崩盘 |
| 用户信任度 | 用户对 AI 输出的信任程度 | 信任一旦失去极难恢复 |
| 输出一致性 | 相同输入得到一致结果的比例 | 用户预期管理的基础 |
| 安全性 | 无有害内容/建议的比例 | 合规底线 |

## 六、自检清单

- [ ] 是否在构建之前定义了"正确"的标准和评判者？
- [ ] 是否有三层防线（定义/验证/监控）？
- [ ] Accept Criteria 是否可以被逐字执行？
- [ ] 产品质量飞轮是否在运转？
- [ ] 评测方法是否分层（代码/AI/人工）？
- [ ] 北极星指标和隐藏指标是否都在被追踪？
