# 04 — 评测框架设计

## 核心原则

> **评测权在业务手里。** 模型团队负责"把分数搞上去"，但"分数衡量什么"的定义权属于 PM。
>
> **你不能评估的东西，你就不应该构建。** — Forbes Tech Council, 2026

## 一、"1+3" 评测法

中国 AI PM 社区 (2025-2026) 沉淀的核心方法论：

```
离线评测（实验室模拟考）
  → 速度快、成本低、可重复
  → 但可能"高分低能"

在线评测/A/B（真实世界高考）
  → 最具说服力
  → 但速度慢、风险高

人工评测 + 红蓝对抗（专家面试 + 压力测试）
  → 弥补自动化盲区
  → 尤其适用安全性和主观体验
```

### 推荐工作流

```
算法提交新模型/新Prompt
  → 离线评测初筛（过基础关）
    → 人工评测（含红军安全测试）
      → A/B 灰度验证
        → 全量上线
          → 持续监控
```

### 各阶段卡点

| 阶段 | 通过标准 | 不通过怎么办 |
|------|----------|-------------|
| 离线评测 | 核心指标不倒退 | 不进入下一阶段 |
| 人工评测 | 安全性、主观质量达标 | 针对性修复 |
| A/B 灰度 | 北极星指标正向 | 回滚、分析、重来 |
| 全量 | 持续监控无异常 | 自动降级 + 人工介入 |

## 二、三层漏斗指标体系

来自腾讯/百度 AI PM 实践：

```
L1 北极星指标（商业目标绑定）
   问题解决率、GMV、用户留存、付费转化
   面向：业务负责人
   规则：一票否决 — 致命错误直接 0 分

  L2 产品/体验指标（用户行为与态度）
     采纳率、任务成功率、交互轮次、满意度
     面向：产品经理
     规则：严重不达标阻塞上线

    L3 模型/技术指标（底层能力）
       准确率、幻觉率、P90 延迟、语气一致性
       面向：算法工程师
       规则：可上线但列入迭代计划
```

### 致命错误清单（L1 一票否决）

- 金额计算错误
- 法律/合规建议错误
- 医疗建议错误
- 隐私数据泄露
- 暴力/歧视/有害内容
- 拒绝服务（该回答的不回答）

### 北极星指标的设计方法

不同产品类型的北极星指标：

| 产品类型 | 北极星指标 | 计算方式 |
|----------|-----------|----------|
| 对话助手 | 周高质量对话数 | WAU × 平均对话轮次 × 有效回答率 |
| RAG 问答 | 首问解决率 | (无追问 + 正反馈) / 总对话数 |
| 内容生成 | 保存发布占比 | 未修改直接发布 / 保存发布总量 |
| 代码助手 | 采纳率 | 被接受的建议 / 全部建议 |
| 客服机器人 | 转人工率 | 转人工对话 / 总对话数（越低越好） |

### 隐藏指标（长期决定生死）

这些指标短期不影响决策，长期跌破则产品崩塌：

| 隐藏指标 | 退化信号 | 恢复难度 |
|----------|----------|----------|
| **用户信任度** | 用户开始验证 AI 的每一个输出 | 极高 — 一旦失去极难恢复 |
| **事实准确性** | Sev-0（看起来对但实际错）比例上升 | 高 — 需要系统性修复 |
| **输出一致性** | 相同问题得到不同回答 | 中 — 破坏用户预期 |
| **安全性** | 有害输出开始出现 | 极高 — 一次重大事故可终结产品 |
| **鲁棒性** | 对抗性输入成功率上升 | 中高 — 需要专门的红队测试 |

## 三、评测集构建

### 四大数据来源

| 来源 | 优势 | 劣势 | 占比建议 |
|------|------|------|----------|
| 线上真实日志 | 反映真实分布 | 标注成本高 | 40% |
| 人工构建 | 质量可控、可覆盖边界 | 成本高、覆盖面有限 | 20% |
| 公开数据集 | 可对比、标准化 | 可能离线分布 | 20% |
| 强模型生成 | 大规模、多样化 | 可能包含幻觉 | 20% |

### 评测集矩阵

| 类型 | 用途 | 更新频率 |
|------|------|----------|
| **通用能力评测集** | 评估整体"平均分"，回答"新模型更好还是更差" | 季度 |
| **细分领域评测集** | 评估特定场景 | 月度 |
| **能力探针评测集** | 测试单一维度（推理、翻译、代码） | 按需 |
| **对抗/安全评测集** | 红军弹药库，安全护栏测试 | 每周更新 |
| **回归评测集** | 历史上出过问题的 Case | 每次出问题就加 |

### 关键原则

> **基于真实失败来定义 Eval，不要坐在房间里脑补。** — Ian Cairns (Freeplay CEO)

"否则你所有的 Eval 都通过，但用户还是不开心。"

**流程：**
1. 从生产环境找 100 个真实 Case，找到真正出问题的
2. 为每个失败类型创建针对性 Eval
3. 持续更新（每次发现新失败模式就加入）
4. 定期淘汰"永远通过"的 Eval（它们没有区分度）

## 四、LLM-as-Judge 的四要素框架

Arize AI (Aman Khan) 的标准结构：

```
┌─────────────────────────────────────────────┐
│ 1. Role — 谁在评价？什么视角？              │
│    "你是一个有 10 年经验的客服质量评估专家"   │
├─────────────────────────────────────────────┤
│ 2. Context — 产品、用户、品牌约束           │
│    "AI 服务企业用户，优先清晰和专业语气"     │
├─────────────────────────────────────────────┤
│ 3. Goal — 具体可衡量的标准                  │
│    "评估准确性、清晰度、可操作性和品牌一致性" │
├─────────────────────────────────────────────┤
│ 4. Labels — 显式评分 Rubric                 │
│    5=Excellent, 3=Acceptable, 1=Failed      │
│    每个级别配具体描述和正反示例              │
└─────────────────────────────────────────────┘
```

### 评分 Rubric 示例

```
5 = Excellent
  - 所有事实准确
  - 回答完整覆盖用户问题
  - 语气符合品牌风格
  - 提供可操作的下一步建议

3 = Acceptable
  - 核心事实准确
  - 回答了用户的主要问题
  - 有 1-2 处小瑕疵（如语气不够一致）

1 = Failed
  - 存在事实错误
  - 遗漏了用户问题的核心部分
  - 语气严重不符合品牌风格
```

### 校准流程

```
1. 人工标注 50-100 个 Case
2. LLM Judge 独立打分
3. 计算一致率（目标 > 85%）
4. 不一致 Case 分析
   ├─ LLM 过严 → 调整 Rubric
   ├─ LLM 过松 → 收紧 Rubric
   └─ LLM 盲区 → 人工 Rule 补充
5. 每两周重复一次
```

## 五、多轮对话评测（5 维度）

传统评测忽略多轮对话的独特性。针对对话式 AI 产品的专属框架：

| 维度 | 评估内容 | 评分标准 |
|------|----------|----------|
| **M1 上下文继承** | 第 N 轮是否利用了前 N-1 轮信息 | 0/1 二分 |
| **M2 指代消解** | "这个""那个"等是否正确指代 | 完全正确 1 / 部分正确 0.5 / 错误 0 |
| **M3 意图切换** | 用户中途换话题是否识别 | 识别并响应 1 / 尚可 0.5 / 死循环 0 |
| **M4 澄清能力** | 模糊提问时是否恰当澄清 | 必要性 × 针对性 |
| **M5 目标达成度** | 对话结束时用户诉求是否解决 | 完全达成 1 / 部分 0.5 / 未达成 0 |

## 六、评测成熟度模型

| 阶段 | 特征 | 标志性能力 |
|------|------|-----------|
| **Lv1 无评测** | 凭感觉判断好不好 | — |
| **Lv2 有评测集** | 有一套测试 Case | 离线评测 |
| **Lv3 自动化** | 每次产出自动评分 | LLM-as-Judge 集成 |
| **Lv4 闭环** | 线上监控 + 定期校准 | 评测→发现→修复→验证闭环 |
| **Lv5 自进化** | 评测集自动更新 | 新失败模式自动发现、自动建评测 |

**80% 的团队在 Lv2-Lv3 之间。**
