# 评测框架体系

## 核心原则

> **评测权在业务手里，不在模型团队手里。**
>
> 模型团队负责"把分数搞上去"，但"分数衡量什么"的定义权必须属于 PM。

## 一、四步评测法

### Step 1：设计评测指标

四级指标同时覆盖：

```
┌──────────────────────────────────┐
│  用户感知层：体验与信任            │
│  相关性 · 完整性 · 语气一致性     │
├──────────────────────────────────┤
│  模型表现层：效果                  │
│  准确率 · 幻觉率 · 鲁棒性         │
├──────────────────────────────────┤
│  系统性能层：速度与稳定性          │
│  P90/P99 延迟 · RPM · 输出速度    │
├──────────────────────────────────┤
│  成本与资源层：预算控制            │
│  单价 · GPU 利用率 · ROI          │
└──────────────────────────────────┘
```

### Step 2：构建评测集

数据来源三合一：
1. **开源基准**（MMLU、GSM-8K 等）— 通用能力基准
2. **线上真实日志** — 反映真实用户分布和场景
3. **人工构建的场景库** — 覆盖长尾场景和边界 Case

评测集的关键特征：
- 覆盖真实的用户行为分布（不是均匀分布）
- 包含对抗性样本（故意刁难的输入）
- 持续更新（每次发现新失败模式都加入评测集）

### Step 3：选择评测方法

分层匹配：

| 方法 | 适用场景 | 成本 | 可靠性 |
|------|----------|------|--------|
| 自动脚本（BLEU/ROUGE 等） | 翻译、FAQ 等"答案明确"任务 | 低 | 中 |
| LLM 评分器 | 语义理解、开放式任务 | 中 | 中高 |
| 人工评测 | 品牌风格、安全性等高敏感维度 | 高 | 最高 |
| 线上 A/B 测试 | 最终效果判断 | 高 | 最高 |

### Step 4：量化与迭代

- 仪表盘实时追踪
- 报警阈值（如幻觉率 >5% 自动降级）
- 每次修复后回归评测集
- 定期更新评测集（反映新失败模式）

## 二、LLM-as-Judge 的正确用法

**不要让模型给自己的产出打分。** 独立验收 Agent 的设计要求：

### 四要素框架（Arize AI）

1. **Judge Role** — 谁在评价？从什么视角？（用户视角/专家视角/安全视角）
2. **Context** — 产品、用户、业务约束、品牌声音是什么？
3. **Clear Goal** — 具体可衡量的标准是什么？
4. **Scoring Labels** — 显式 Rubric：

```
5 = Excellent  — 完美满足标准，超出预期
4 = Good       — 满足全部标准
3 = Acceptable — 基本满足，有小瑕疵
2 = Poor       — 部分不满足，有明显问题
1 = Failed     — 完全不符合标准
```

### 校准流程

1. 人工标注 50-100 个 Case
2. 让 LLM Judge 独立打分
3. 计算人工 vs LLM 的一致率（目标 >85%）
4. 不一致 Case 分析 → 调整 Rubric
5. 定期重新校准

## 三、北极星指标 vs 隐藏指标

### 北极星指标（直接驱动决策）

- 首问解决率：AI 首次交互就解决用户问题的比例
- 高质量对话数：被用户标记为有用的对话频次
- 任务完成率：用户成功完成目标任务的比例

### 隐藏指标（长期决定生死）

这些指标短期不直接影响决策，但长期跌破阈值会导致产品崩塌：

- **用户信任度** — 一旦失去，极难挽回
- **事实准确性** — Sev-0 错误（看起来对但实际错）是最危险的
- **安全性** — 一次重大安全事故可能终结产品
- **输出鲁棒性** — 对抗性输入下的表现

### 监控策略

| 指标类型 | 监控频率 | 报警阈值 | 响应动作 |
|----------|----------|----------|----------|
| 北极星 | 实时 | 下降 10% | 排查 + 灰度回滚 |
| 隐藏-准确性 | 每日抽样 | 幻觉率 >5% | 降级模型 |
| 隐藏-安全 | 实时 | 任一 Sev-0 | 立即下线 |
| 性能 | 实时 | P99 > 阈值 | 扩容/降级 |

## 四、评测体系成熟度模型

| 阶段 | 特征 | 下一步 |
|------|------|--------|
| Lv1 无评测 | 凭感觉判断 AI 产出好不好 | 建立基础评测集 |
| Lv2 有评测集 | 有一套测试 Case | 引入 LLM-as-Judge |
| Lv3 自动化评测 | 每次产出自动评分 | 建立评测校准机制 |
| Lv4 持续评测 | 线上实时监控 + 定期校准 | 建立反馈闭环 |
| Lv5 自进化评测 | 评测集自动更新 + 新失败模式自动发现 | — |

## 五、自检清单

- [ ] 是否有四层评测指标？
- [ ] 评测集是否融合了开源基准 + 线上日志 + 人工场景？
- [ ] LLM Judge 是否独立于执行 Agent？
- [ ] 是否有定期校准流程（人工 vs LLM 一致率）？
- [ ] 北极星和隐藏指标是否都被监控？
- [ ] 是否有报警和自动响应机制？
