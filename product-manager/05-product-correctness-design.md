# 05 — 产品正确性设计

## 核心问题

> 怎样设计产品来保证正确性？如何确保最终结果是更好的用户体验？

## 一、四层防线模型

```
第 0 层：定义正确性（构建之前）
  "什么是正确？谁来判断？标准是什么？证据是什么？"

第 1 层：设计约束（构建之中）
  "通过设计缩小错误可能性空间"

第 2 层：多层验证（构建之后、上线之前）
  "代码层 → AI 层 → 人工层，逐层把关"

第 3 层：生产监控（上线之后）
  "上线不是结束，是大规模测试的开始"
```

### 第 0 层：定义正确性

在开始构建之前必须回答的 4 个问题：

| 问题 | 需要明确 | 反面案例 |
|------|----------|----------|
| **谁是裁判？** | 用户？PM？独立 Agent？组合？ | "大家看看觉得行就行" |
| **标准是什么？** | 精确的 AC + 可量化的指标 + Rubric | "做到让用户满意" |
| **证据是什么？** | 测试结果？用户行为？A/B 数据？ | "代码看起来对的" |
| **失败会怎样？** | 用户流失？错误决策？法律责任？ | 没想过会失败 |

### 第 1 层：设计即是约束

好的产品设计通过**缩小 AI 的可能性空间**来保障正确性：

| 约束手段 | 作用 | 实例 |
|----------|------|------|
| **Design System** | 限制组件选择，防 UI 漂移 | "只能用这 24 个组件" |
| **模板化交互** | 限制交互模式，保一致性 | "表单只有这 3 种布局" |
| **结构化输出** | 限制 AI 输出格式 | JSON Schema 约束 |
| **类型系统** | 编译期排除一大类错误 | TypeScript strict mode |
| **预设边界** | 明确 AI 不能做什么 | "不要给医疗建议" |
| **输入校验** | 在入口过滤问题输入 | SQL 注入检测、长度限制 |

### 第 2 层：多层验证

```
┌────────────────────┐
│  人工审查（顶层）    │  ← 抽样审计 + 高风险决策 + 校准基准
├────────────────────┤     频率：每日/每周
│  AI 验证（中层）    │  ← LLM-as-Judge + 幻觉检测 + 安全扫描
├────────────────────┤     频率：每次产出
│  代码级检查（底层）  │  ← 测试 + Lint + 类型 + Schema + 视觉回归
└────────────────────┘     频率：每次产出
```

### 第 3 层：生产监控

| 监控指标 | 报警阈值 | 响应 |
|----------|----------|------|
| 幻觉率 | > 5% | 自动降级模型 |
| Sev-0 错误 | 任意一个 | 立即下线该功能 |
| 用户挫败率（重复操作+跳出） | 环比上升 20% | 排查 + 灰度回滚 |
| P99 延迟 | > 3s | 扩容或降级 |
| 转人工率 | 环比上升 30% | 检查 AI 质量 |

## 二、AC 即功能规格

### 原则

> AC（Accept Criteria）不再是团队内的沟通辅助，而是会被 AI 逐字执行的功能规格说明书。

### 对比

**差的 AC：**
```
"用户可以筛选搜索结果"
```

**好的 AC（AI 可执行）：**
```
筛选面板
├─ 位置：搜索结果列表上方，position: sticky
├─ 筛选项：
│   ├─ 类别（多选 checkbox，最多选 5 个）
│   ├─ 价格范围（双滑块，min/max 动态从结果集计算）
│   └─ 评分（单选 radio，1-5 星）
├─ URL 持久化：筛选参数反映在 query string，支持分享和后退
├─ 空态：无匹配结果时显示「尝试放宽筛选条件」+ 建议的替代筛选
├─ 加载态：骨架屏（非空白），筛选后 200ms 内出现
├─ 错误态：网络失败时保留上次结果 + 「重试」按钮
└─ 移动端：底部抽屉面板，含「应用」和「重置」按钮，滑动手势关闭
```

### 自测方法

拿出最近 10 个 Ticket，假设读者：
- 从未见过你的产品
- 从未和你的用户交谈
- 会逐字遵守每一条

如果 AC 在这些条件下仍然成立 — 合格。

## 三、失败模式设计

### 写入失败 Spec

传统 PRD 只写"产品应该做什么"。AI 时代需要同时写"产品绝对不能做什么"：

```
失败 Spec 模板：
├─ 绝对不能做的（Sev-0 — 立即下线）
│   ├─ 给出可能造成人身伤害的建议
│   ├─ 泄露用户个人信息
│   └─ 在金融/医疗/法律场景给出确定性错误答案
├─ 不应该做的（Sev-1 — 紧急修复）
│   ├─ 在没有足够信息时做出确定性判断
│   ├─ 混用不同用户的上下文
│   └─ 生成不可操作/无意义的回答
└─ 尽量避免的（Sev-2 — 排期修复）
    ├─ 语气与品牌调性不一致
    └─ 过度使用技术术语
```

### 失败恢复

在产品设计中内置恢复机制：

| 机制 | 作用 | 实例 |
|------|------|------|
| **撤销** | 操作可逆 | "修改由 AI 建议，点击采纳/撤销" |
| **回滚** | 版本可回退 | 每次上线保留上一版本，一键回滚 |
| **置信度标记** | 用户知道哪些是推断 | "这可能不准确"的标记 |
| **二次确认** | 关键操作前确认 | "AI 建议删除 XX，确认？" |
| **安全出口** | 总能回到安全状态 | "重置为默认设置" |
| **降级模式** | AI 不可用时仍可用 | 基础功能不依赖 AI |

## 四、渐进式暴露 vs 校准式协作

### 原则 A：渐进式暴露

不要让用户一次性面对 AI 的全部能力：

```
Level 1 新手模式
  └─ 有限选项、高引导、更多约束

Level 2 熟练模式
  └─ 更多控制、更少引导

Level 3 专家模式
  └─ 完整 API、自定义配置、高级功能
```

### 原则 B：校准式协作

AI 产品设计的目标不是 100% 自动化，而是找到**最优人机协作点**：

| 决策类型 | 协作模式 | 示例 |
|----------|----------|------|
| 确定性高、风险低 | AI 自动执行 | 格式转换、排序、分类 |
| 确定性中、风险中 | AI 建议 + 人工确认 | 内容生成、翻译 |
| 确定性低、风险高 | 人工决策 + AI 辅助 | 战略决策、创意方向 |
| 涉及价值观/法律 | 纯人工 | 品牌定义、合规判断 |

## 五、质量飞轮

来自 O'Reilly Radar / TestRail：

```
Offline Quality（上线前 — 风洞测试）
  → 定义参考示例、指标、Rubric
    → Online Quality（上线后 — 赛道监控）
      → 用户显/隐性信号 → 发现异常
        → Continuous Improvement（事后 — 采样失败）
          → 发现新的失败模式
            → Update Evals（更新评测体系）
              → 更新指标 & 参考数据 & 修复
                → 回到 Offline Quality（新一轮）
```

核心 Insight：
> "生产环境会超越你所有的最佳努力。无论上线前的指标多好，实际使用中总会出现意外。"

## 六、用户体验的三层设计

来自 2025-2026 中国 AI PM 社区：

| 层级 | 设计重点 | 衡量指标 | 实例 |
|------|----------|----------|------|
| **效率层** | 减少操作步骤、降低延迟 | 任务完成时间、P90 延迟 | 一键生成报告 |
| **协作层** | 人机协作工作流 | 协作满意度、采纳率 | AI 建议 + 人工修改 |
| **创造层** | AI 辅助创意、个性化 | 增量价值、用户惊喜度 | AI 生成的独特内容 |

### 体验自检

- [ ] 效率层 — P90 < 800ms？操作步骤是否最少？
- [ ] 协作层 — 用户在 AI 产出上修改了多大比例？（太多 = AI 没帮上忙，太少 = 用户盲信）
- [ ] 创造层 — 是否有用户"没想到但很喜欢"的瞬间？

## 七、自检清单

- [ ] 四层防线（定义/设计约束/验证/监控）是否完整？
- [ ] AC 是否可以被零上下文的人逐字执行？
- [ ] 是否写了失败 Spec（不能做什么）而不仅仅是功能 Spec（要做什么）？
- [ ] 产品是否有恢复机制（撤销/回滚/降级/安全出口）？
- [ ] 人机协作模式是否按风险分层？
- [ ] 质量飞轮是否在持续转动？
- [ ] 三层层体验（效率/协作/创造）是否都在被关注？
