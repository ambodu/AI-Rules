# 09 — UX 北极星：把用户放回 OS 的中心

## 为什么需要这份文档

v3 的 AI Product OS 描述了完美的内核架构——但**一次都没提到用户体验**。

用户的核心目标是"更好的用户体验"。如果整套方法论不能让用户体验变好，它就是一套精美的废纸。

## UX 在 AI Product OS 中的位置：Display Server

在 Linux 中，Kernel 管理硬件，X11/Wayland 管理用户看到的东西。**Kernel 不关心用户体验——Display Server 关心。**

AI Product OS 同样需要这个层：

```
AI Product OS 完整层次：

┌──────────────────────────────────────────┐
│           USER EXPERIENCE LAYER          │  ← 本应存在但 v3 缺失
│  UX Metrics / UX Policy / UX Verification│
├──────────────────────────────────────────┤
│           USERSPACE (Agents)             │
├──────────────────────────────────────────┤
│           KERNEL (Gate/Scheduler/...)     │
└──────────────────────────────────────────┘
```

## 一、UX 如何嵌入每个 Kernel 子系统

### VERIFY Gate 的 UX 维度

当前 L1-L4 Gate 检查的是：格式、安全、AC 符合度。**缺了 UX 检查。**

```
现有 L3 检查:                  应该增加的 UX L3 检查:
├─ 是否符合 AC?                ├─ 用户能看懂这个界面吗?
├─ 是否有幻觉?                 ├─ 完成这个任务需要几步?
├─ 是否安全?                   ├─ 有没有让用户困惑的地方?
└─ 评分 1-5                    └─ 有没有不必要的认知负荷?
```

**具体做法：** 在 L3 验收 Prompt 中加入 UX 评估维度。
- "这个界面是否清晰表达了当前状态?"
- "用户在最常见的场景下需要点几次才能完成任务?"
- "有没有信息过载或选项过载?"

### SCHEDULER 的 UX 维度

当前 MLFQ 按 Bug 严重度调度。**缺了 UX 优先级。**

| 当前 P 级 | 应该加入 UX 因素 |
|-----------|-----------------|
| P0 = Sev-0 缺陷 | P0 也应该包括"大量用户困惑/投诉/流失" |
| P1 = Sev-1 缺陷 | P1 也应该包括"核心体验路径上的摩擦" |
| P2 = Sev-2 缺陷 | P2 也应该包括"次要体验问题" |

**"用户挫败"应该是调度器的第一优先级信号，和 Sev-0 同级。**

### MEMORY 的 UX 维度

当前 MEMORY 存储 AC、Eval 结果、决策历史。**缺了 UX 知识。**

新增 UX Memory Tier：
```
UX Hot:  当前用户的 session 上下文（从哪来的、在做什么、卡在哪）
UX Warm: 最近发现的 UX 问题模式（"搜索功能用户重复操作率上升"）
UX Cold: 用户研究归档、Usability Test 结果、长期 UX 指标趋势
```

## 二、UX 作为反馈机制的最终锚点

v3 的反馈机制面临"谁来验证验证者"的无限递归。**UX 指标终止这个递归。**

```
验证链：
  L1 检查 → L2 检查 → L3 AI 验收 → L4 人工审查
    ↑                                    │
    └──────── 谁验证这些是对的? ──────────┘
                        │
                        ▼
              ┌─────────────────┐
              │   UX 终极锚点    │
              │                 │
              │ . 用户任务完成率 │
              │ . 用户满意度 CSAT│
              │ . 用户留存/回访  │
              │ . 用户挫败信号   │
              │                 │
              │ 如果 L1-L4 全部 │
              │ PASS 但用户不用 │
              │ → 反馈机制本身  │
              │   有问题        │
              └─────────────────┘
```

**判断反馈机制是否正确的方法**：看最终的 UX 指标。

如果所有 Gate 都通过、所有 Eval 都 PASS、所有 Watchdog 都静默——但用户不用你的产品，或者用了很痛苦——那你的反馈机制是错的。不管你做了多少三角验证和定期校准。

## 三、UX 驱动的产品正确性

v3 的产品正确性公式：`CAPABILITY + POLICY + CHECKPOINT`

**缺少了 UX 因子。** 更完整的公式：

```
产品正确性 = (CAPABILITY + POLICY + CHECKPOINT) × UX_ALIGNMENT

其中 UX_ALIGNMENT 的取值范围是 0 到 1。
如果功能正确但用户不会用 → UX_ALIGNMENT = 0 → 整体正确性 = 0
```

### UX Alignment 的评估

| 等级 | UX_ALIGNMENT | 表现 |
|------|-------------|------|
| 0.0 | 用户无法完成核心任务 | 功能存在但不可用 |
| 0.3 | 能完成但极痛苦（多次试错） | 只有 power user 能坚持 |
| 0.6 | 能完成，有一些摩擦 | 需要进一步优化 |
| 0.8 | 流畅完成，基本无摩擦 | 已达到良好用户体验 |
| 1.0 | 用户觉得比预期更好（愉悦） | 超出预期的体验 |

**UX_ALIGNMENT 只能从用户行为数据中获得，不能从 Gate 验证中获得。**

## 四、UX 信号采集

v3 关注的是系统信号（测试结果、Eval 分数、预算消耗）。需要同等重视 UX 信号：

| UX 信号 | 采集方式 | 健康阈值 |
|---------|----------|----------|
| **任务完成率** | 用户漏斗分析 | > 80% |
| **操作步数** | 完成任务的平均操作数 | 不上升 |
| **重复操作率** | 同一操作 30s 内重复 | < 5% |
| **求助率** | 使用帮助/客服的次数 | 不上升 |
| **用户反馈情感** | 反馈内容的情感分析 | 负面 < 10% |
| **PES (感知努力度)** | 任务后问卷 | < 3/7 ("非常容易"到"非常困难") |
| **SUS (系统可用性量表)** | 标准 SUS 问卷 | > 68 (行业平均) |
| **NPS** | 标准 NPS 问卷 | > 30 |

## 五、UX Gate：第 5 层验证

在 L1-L4 之外增加一个 UX Gate：

```
L5: UX Gate (混合确定性+概率性)
├─ UX Lint: 自动检查 UX 规则
│   · 点击区域是否 >= 44x44px (移动端)?
│   · 文本对比度是否 >= 4.5:1?
│   · 是否有超过 7 个主导航项?
│   · 移动端是否有水平滚动?
│   → PASS/FAIL: 基于规则的 UX Lint
│
├─ UX LLM Judge: AI 评估 UX 质量
│   · "模拟一个急躁的用户，评估这个界面的信息优先级"
│   · "这个流程中有没有不必要的步骤?"
│   · "用户在这里最可能犯什么错误?"
│   → SCORE 1-5: 基于 UX Rubric
│
└─ UX HITL: 高风险 UX 决策
    · 彻底改版的首次发布
    · AI 生成的用户流程
    · 面向弱势群体的产品
    → 必须通过 Usability Test (5+ 真实用户)
```

## 六、Gate 通过 ≠ UX 好：警惕"Gate 幻觉"

最大的风险：系统显示所有 Gate 都是绿色，所有指标都正常，你就以为产品很好。

**这是一个致命的幻觉。** Gate 能告诉你"没有明显的错误"。Gate 不能告诉你"用户喜欢这个产品吗"。

```
Gate 全部 PASS 时 PM 应该问自己的问题：

1. 上一次和真实用户交谈是什么时候？
2. 用户实际使用这个功能的方式是我们预期的吗？
3. 有没有用户用得很痛苦但从没反馈的？
4. 我们的 Gate 是不是在度量"容易度量的东西"而非"对用户重要的东西"？
5. UX_ALIGNMENT 是在上升还是下降？
```

## 七、UX 驱动的迭代节奏

v3 的周度 Eval 会议关注的是：发现了哪些新失败模式？Eval 是否需要更新？

**加一个 UX 专属议程：**

```
每周 UX Pulse (30 min):
1. 本周 UX 信号回顾 (5 min)
   - 任务完成率趋势
   - 挫败信号趋势
   - 用户反馈情感趋势

2. 最痛苦的 3 个用户场景 (10 min)
   - 看 3 段真实用户 session 回放
   - 不看数据看实际行为

3. UX Gate 通过率 vs 用户满意度差异 (10 min)
   - Gate 说 PASS 但用户不满意的案例
   - Gate 说 FAIL 但用户其实能接受的案例
   - 为什么会有这些差异？

4. 这周我们需要修正的 UX 问题 (5 min)
   - 优先级排序
   - 责任人
```

## 八、最后的矫正

v3 说：**"AI PM 的工作是设计一个操作系统，让 AI 在里面安全地做产品。"**

这句话对了一半。完整版应该是：

**"AI PM 的工作是设计一个操作系统，让 AI 在里面安全地做产品 —— 然后确保这个产品真的让用户的生活变得更好了。"**

Kernel 设计得再好，如果用户不快乐，一切都是零。
