## 二、周度工作节奏

```
周一上午 — 定义
  · 上传项目 Brief → AI 生成结构化 PRD
  · PM 审查、优化、补充 AC、批准
  · 产出：可被 AI 执行的 PRD + AC

周一下午 — 启动
  · PRD 进入开发流水线
  · AI Agent 开始生成代码
  · PM 同步本周优先级给团队

周三上午 — 首次审查
  · 审查 AI 初始产出
  · 对照 AC 逐项检查
  · 产出：偏差列表 + 需要修正的方向

周三下午～周四 — 迭代
  · 收紧偏离的 Spec（用更精确的语言重写 AC）
  · 确认达标的产出
  · 标记需要工程师技术审查的部分
  · 产出：可验收的产出

周五上午 — 验收 + 周度 Eval 会议
  · 最终验收通过
  · 周度 Eval 会议（4 个核心问题）
  · 清理评测集（淘汰"永远通过"的 Case）

周五下午 — 展望
  · 定义下周工作
  · 回顾评测数据趋势
```

### 关键变化

| 过去的节奏 | 现在的节奏 |
|-----------|-----------|
| 状态同步会 | **取消** — 产出几天内可见 |
| 对齐会 | **减少** — 精确的 AC 减少理解偏差 |
| 估点/排期 | **简化** — 大部分工作在几天内完成 |
| 技术评审会 | **后移** — PM 先做意图审查，再技术评审 |

### 周度 Eval 会议的 4 个核心问题

来自 Productboard/Freeplay 最佳实践：
1. 本周发现了哪些新的失败模式？
2. 哪些 Eval 本该捕获但没捕获？（Eval 的盲区）
3. 哪些 Eval 是"永远通过"的 — 它们是否还有区分度？
4. 下周需要新增哪些 Eval？

## 三、PM 审查清单

PM 不做代码审查（那是工程师的事），PM 做**产品意图审查**：

### 每次审查必问

1. **意图匹配** — AI 的产出是否在做我们让它做的事？
2. **边缘情况** — 指定的空/错误/加载/边界状态是否都处理了？
3. **用户目标** — 是否真正解决用户的问题（而非仅仅"技术上正确"）？
4. **范围控制** — AI 是否偷偷加了不该加的东西？（Scope Creep 检测）
5. **体验一致性** — UI/交互是否符合 Design System？
6. **伦理安全** — 是否可能造成伤害或误导？

### 信任建立机制

| 做法 | 原因 |
|------|------|
| PR 保持小（3-7 屏 diff） | 越小越易审查，风险越低 |
| 亲手写 PR 标题和描述 | AI 总结"做了什么"，人需要"为什么" |
| 附带测试截图/录屏 | 证据 > "代码看起来对" |
| 标注 AI 使用范围 | 信号：哪些经人工审查、哪些纯 AI |
| 提交前自查 | 展现人类责任感，提前发现明显问题 |

### 审查反模式

| 反模式 | 表现 | 纠正 |
|--------|------|------|
| **橡皮图章** | AI 建议接受率 > 85%，从不质疑 | 强制至少找出一个问题 |
| **重写冲动** | 看到不完美就想自己改 | 判断：是否真的影响用户？ |
| **完美主义瘫痪** | 永远觉得还能更好，从不 Approve | 设定明确的时间盒和及格线 |

## 四、失败案例：钉钉 ONE 项目

2025 年广泛传播的复盘（7.5 万字离职长文）揭示了四个关键教训：

### 教训 1：用户定义未闭环

在"为员工服务"和"为管理者服务"之间持续摇摆。两个用户群体有直接利益冲突，但没有做出明确的取舍。结果是双方都不满意。

**对 AI PM 的启示：** AI 工具不是中立的。它会放大选择。在开始构建前必须明确：为谁服务？不为谁服务？

### 教训 2：未明确保护对象

AI 的引入放大了已有的权力结构。没有明确谁是被保护的对象（员工？管理者？组织？），AI 默认会服务于更有权力的一方。

**对 AI PM 的启示：** 在设计 AI 产品时，必须明确"AI 在保护谁的利益"。这不是技术问题，是产品伦理问题。

### 教训 3：反馈来源错位

"敏捷"的迭代方向被"向老板汇报"而非"用户反馈"驱动。每一次 Sprint Review 面向的是管理者而非实际使用产品的员工。

**对 AI PM 的启示：** 反馈回路必须连接到真正的终端用户。如果你的 Eval 在关注老板关心的指标而非用户关心的指标，产品会偏离。

### 教训 4：效率收益归属不明确

AI 省下的时间归谁？如果归管理者，则变成监控工具被员工抵制；如果归员工，则变成效率工具被欢迎。

**对 AI PM 的启示：** AI 产品设计必须回答"效率收益归谁"——这直接决定产品被欢迎还是被应付。

## 五、Builder-PM 实操：4 小时原型工作流

来自 Forbes (2026.06) 和 FourWeekMBA：

```
Hour 1：定义 + 生成
  · 写 200 字的问题描述 + 用户场景
  · 用 Claude Code / Cursor 生成初始版本
  · 产出：可运行但粗糙的 MVP

Hour 2：迭代 + 修复
  · 试用 → 发现 3 个最大的问题
  · 逐个修复（AI 辅助）
  · 产出：解决了核心问题的 V2

Hour 3：打磨 + 数据
  · 接入最小埋点
  · 添加基础 UI（Design System 组件）
  · 产出：可展示的 Demo

Hour 4：验证 + 文档
  · 对着原问题试用，检查是否真正解决了
  · 记录已知问题和下一步
  · 产出：可演示的原型 + 1 页概述
```

## 六、最小可行 AI PM 系统

如果你今天就要开始：

```
第一周：
├─ Day 1-2：建立基础评测集（至少 50 Case）
├─ Day 3：配置一个 LLM-as-Judge（独立模型）
├─ Day 4：写一份 AC 模板（零歧义标准）
└─ Day 5：部署基础监控（幻觉率 + 用户挫败信号）

第二周：
├─ 开始周度 Eval 会议
├─ 搭建反馈分类管道
├─ 建立校准流程（人工 vs AI）
└─ 启用独立验收 Agent

第一个月：
├─ 评测体系覆盖所有核心场景
├─ 三层漏斗指标完整运转
├─ 质量飞轮开始转动（至少转 2 轮）
└─ 团队适应新节奏
```

## 七、评估 AI PM 系统本身是否有效的指标

| 元指标 | 计算方法 | 健康值 |
|--------|----------|--------|
| **Review-to-Fix 比率** | AI 评审发现的问题中，有多少被实际修复 | > 70% |
| **Eval 新鲜度** | 最近 30 天更新的 Eval 占比 | > 30% |
| **校准偏移** | 本月 vs 上月人工/AI 一致率 | 不应持续下降 |
| **反馈闭合率** | 反馈 → 修复 → 验证完成的闭环比例 | > 80% |
| **审查疲劳度** | 审查者 Pass 率 | 70-85%（<70% 太严，>85% 太松）|
| **逃逸率** | 上线后发现的 Sev-0/Sev-1 数量 | 持续下降趋势 |

## 八、自检清单

- [ ] 工具栈是否覆盖定义→评测→反馈→验证四层？
- [ ] 是否有固定的周度 Eval 会议和议程？
- [ ] PM 是否在做产品意图审查（而非代码审查）？
- [ ] 是否有审查反模式的检测机制？
- [ ] 反馈管道是否闭环（信号→处理→验证→记忆）？
- [ ] AI PM 系统本身的效果是否在被衡量（元指标）？
- [ ] 团队是否在削减会议并增加定义和审查的时间？
