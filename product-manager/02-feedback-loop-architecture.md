# 02 — 反馈回路架构

## 核心问题

> 什么样的反馈机制设计，能保证反馈本身正确，最终确保产品正确？

## 一、三层架构：Model → Harness → Loop → Memory

"人人都是产品经理"社区 (2025-2026) 提出的 AI 产品分层模型：

```
Layer 1: Model（引擎）
  └─ 基础推理能力、训练数据、模型参数

Layer 2: Harness（车间 — 静态环境）
  └─ 工具调用、权限系统、沙箱隔离、API 接入

Layer 3: Loop（排班与验收制度 — 运行动态）
  └─ 触发条件、反馈收集、质量判定、重试策略、停止条件
  
Layer 4: Memory（记忆 — 跨循环学习）
  └─ 错误记录 → 根因诊断 → 验证诊断 → 提炼规则 → 未来复用
```

### Loop 的核心运行逻辑

```
模型执行一轮
  → 从环境收到反馈（测试结果 / 用户信号 / 独立评审）
    → 对照验收标准检查
      → 达标 → 输出，写入 Memory
      → 未达标 → 带着具体反馈进入下一轮
        → 达到最大迭代次数仍不达标 → 人工介入
```

### Loop 设计的 4 个关键问题

1. **谁判定完成？** — 单一 Agent？独立 Agent 委员会？人工？组合？
2. **反馈从哪里来？** — 测试结果？用户行为？独立评审 Agent？三者都要？
3. **什么时候停？** — 最大迭代次数？预算上限？质量阈值？时间限制？
4. **记什么、何时用？** — 只记失败 Case？记成功的模式？记决策理由？

## 二、反馈闭环的三大环节

### 环节 1：信号收集

| 信号类型 | 来源 | 特点 | 处理难度 |
|----------|------|------|----------|
| 显性反馈 | 评分、纠错、投诉、NPS | 质量高、量少 | 低 |
| 隐性反馈 | 停留时长、跳出、转人工、重复操作 | 量大、需解读 | 中 |
| 专业反馈 | 安全审查、品牌审查、专家标注 | 质量最高、成本最高 | 中 |
| 系统反馈 | 测试结果、Lint、性能指标、幻觉检测 | 自动化、即时 | 低 |

**关键原则**：显性反馈告诉你"哪里出了问题"，隐性反馈告诉你"哪里让用户不爽"，系统反馈告诉你"哪里没通过底线"。

### 环节 2：信号处理

#### 分类标注 — 四类问题

| 问题类型 | 根因 | 修复路径 |
|----------|------|----------|
| 功能性缺陷 | AI 输出结果错误 | Prompt 优化 / 模型微调 |
| 体验问题 | 交互不流畅、信息架构混乱 | 产品设计调整 |
| 内容错误 | 知识过时或幻觉 | RAG 知识库更新 |
| 安全风险 | 有害输出、注入攻击、隐私泄露 | 安全护栏 + 模型降级 |

#### 严重度分级（Microsoft/Enterprise AI 标准）

| 级别 | 定义 | 用户影响 | 响应 |
|------|------|----------|------|
| **Sev-0** | 答案看起来正确但实际错误 | 侵蚀信任 — **最危险** | 立即下线 |
| **Sev-1** | 答案明显错误，用户无法自行恢复 | 挫败用户 | 紧急修复 |
| **Sev-2** | 答案明显错误，用户可重新表述恢复 | 惹恼用户 | 排期修复 |

#### 优先级公式

```
处理优先级 = 严重度 × 影响用户数 × 频率 × 战略重要性
```

### 环节 3：效果验证

| 验证方式 | 适用场景 | 速度 | 可信度 |
|----------|----------|------|--------|
| 离线评测 | 每次修改后必做 | 分钟级 | 中 — "高分低能"风险 |
| 灰度发布 | 中等风险修改 | 小时级 | 中高 |
| A/B 测试 | 用户体验相关修改 | 天级 | 高 |
| 持续监控 | 所有已上线功能 | 实时 | 中 — 发现异常但无法归因 |

## 三、数据飞轮：百度开发者社区方法论

### 六步链路

```
数据采集
  → 数据清洗与结构化
    → 模型训练/微调
      → A/B 测试验证
        → 产品功能迭代
          → 用户增长 → 更多数据采集（飞轮加速）
```

### 飞轮启动的最小可行条件

1. **有用户** — 至少 100 DAU，否则数据量不足以驱动飞轮
2. **有埋点** — 关键行为都打点（点击、跳出、转人工、完成）
3. **有评测基准** — 飞轮转一次前后的效果可以对比
4. **有归因能力** — 效果好/坏分别是因为什么

### 冷启动阶段的做法

在用户量不足以驱动飞轮时：
- 人工构建黄金评测集
- 用强模型（GPT-4/Claude Opus）生成多样化评测样本
- 内部 Dogfood 并记录所有异常
- 将每一次 Bad Case 转化为评测样本

## 四、反馈频率矩阵

| 反馈层级 | 频率 | 触发方式 | 反馈内容 | 响应方 |
|----------|------|----------|----------|--------|
| **实时** | 每次输出 | 自动 | Lint、格式、安全扫描、幻觉检测 | 系统 |
| **短期** | 每任务/每日 | 自动 + 手动 | AI 验收 Agent 评审 | 执行 Agent + PM |
| **中期** | 每周 | 手动 | PM 意图审查 + 用户测试 + Eval 周会 | PM + 工程师 |
| **长期** | 每月/每版本 | 手动 | 数据分析 + 战略对齐 + 评测体系更新 | PM + 管理层 |

### 周度 Eval 会议的核心议程

来自 Productboard/Freeplay 的最佳实践：
1. 本周发现了哪些新的失败模式？
2. 哪些 Eval 本该捕获但没捕获？
3. Eval 中有哪些是"永远通过"的——它们是否还有意义？
4. 下周需要新增哪些 Eval？

## 五、反馈质量的自我保障

### 五个保障机制

1. **三角验证** — 任一质量判断需要至少两个独立信号源确认
   - 例：AI Judge 说 Pass + 用户行为数据无异常 = 真 Pass
   - 例：AI Judge 说 Pass + 用户跳出率激增 = 假 Pass，需调查

2. **定期校准** — 每两周一次：人工评估 vs AI 评估的一致率
   - 一致率目标 > 85%
   - 不一致 Case 分析 → 找到 Rubric 的盲区 → 更新

3. **疲劳检测** — 监控审查质量
   - AI 建议接受率 > 85% → 审查失效预警
   - 审查者产出速度异常提升 → 可能在跳读
   - 同一审查者 Pass 率持续 100% → 可能在偷懒

4. **假阳性/假阴性审计** — 每月一次
   - 抽查 50 个 Pass 的产出，是否存在漏检缺陷（假阴性）
   - 抽查 50 个 Fail 的产出，是否属于误杀（假阳性）

5. **Priority Waterfall** — 反馈处理顺序
   - Sev-0（看起来对实际错）> Sev-1（明显错且阻塞）> Sev-2（明显错可恢复）
   - 修复的严重度升级 > 新增功能
   - 信任修复 > 体验优化 > 新功能
